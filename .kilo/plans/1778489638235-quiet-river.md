# Plan: SiteBuilder Form Alignment with New Data JSON Format

## Context

Ramen Taro's `data.json` and `data-seo-example.json` define the new canonical data format. The `/sitebuilder` form (`SiteBuilderForm.tsx`), `SiteBuilderData` type (`storage.ts`), and `mapBuilderToDataJson` mapper (`site-mapper.ts`) need to be aligned so data round-trips correctly between the form and the JSON output.

## Analysis Summary

The current form and mapper are **missing several fields** and have **structural mismatches** compared to the target JSON format. Below is a detailed breakdown.

---

### 1. Missing Top-Level Fields (ramen-taro `data.json`)

| Field in data.json | Present in form/mapper? | Action Needed |
|---|---|---|
| `uid` | ❌ No | Add to `SiteBuilderData`, form defaults, and mapper |
| `version` | ❌ No | Add to `SiteBuilderData`, form defaults, and mapper |
| `tagline` | ❌ No | Add to `SiteBuilderData`, form, and mapper |

### 2. Missing/Incomplete `pages` Structure

The `data.json` uses a **block-based page schema** (`pages.home`, `pages.about`, `pages.menu`, `pages.contact`) with `sections` arrays containing `id`, `type`, `ui`, and `data` objects. The SiteBuilder **does not expose page structure editing at all**. The generated `data.json` from the mapper has **no `pages` field**.

**Action:** The mapper needs to auto-generate the `pages` structure from form data, OR a new "Pages" step must be added. Auto-generation is recommended for MVP since the form already captures the data needed for each section.

### 3. `contact` Structure Mismatch

`data.json` nests location-like fields under `contact`:
```
contact.address, contact.phone, contact.email, contact.location.{lat,lng,mapsUrl,embedUrl}, contact.openingHours[], contact.holidayNotes
```

The mapper currently outputs `contact` with `address`, `phone`, `email` but puts `location`-like fields at the top level (`location.lat`, `location.lng`, etc.) — **not** nested under `contact.location`.

**Action:** Update mapper to output:
```
contact: {
  address, phone, email,
  location: { lat, lng, mapsUrl, embedUrl, address, plusCode },
  openingHours,
  holidayNotes
}
```

### 4. `schema.aggregateRating` — Not Mapped

`data.json` includes:
```
schema.aggregateRating: { ratingValue, reviewCount, bestRating, worstRating, source, sourceUrl }
```
The mapper does **not** produce `aggregateRating`.

**Action:** Add fields to `SiteBuilderData` schema section (ratingValue, reviewCount, bestRating, worstRating, source, sourceUrl) and mapper output.

### 5. `images.logo.id` — Not Mapped

`data.json` logo has an `id` field. Mapper hardcodes url to `/images/logo.png`.

**Action:** Add optional `logoId` to `SiteBuilderData` and include in mapper.

### 6. Opening Hours Day Format

`data.json` uses abbreviated: `"Mon - Thu"`, `"Fri - Sat"`, `"Sun"`.
Form defaults use full names: `"Monday - Thursday"`, `"Friday - Saturday"`, `"Sunday"`.
`data-seo-example.json` uses `"Public Holidays"` with a `notes` field and `isClosed` flag.

**Action:** Standardize to abbreviated format in form defaults and mapper. Add optional `isClosed` and `notes` fields to opening hours.

### 7. Menu Item Enrichment

`data-seo-example.json` menu items have rich fields not in the form:
- `id`, `secondaryName`, `isVegetarian`, `isVegan`, `isSpicy`, `spiceLevel`, `allergens[]`, `calories`, `ingredients[]`, `available`, `availableFrom`, `availableTo`, `size`, `limited`, `availableUntil`, `price` as string with currency symbol (e.g., `"¥1,500"`)

The form uses `price` as a plain number and only captures `name`, `description`, `price`, `category`, `isPopular`, `image`.

**Action:** Add enrichment fields to menu item form and mapper. Store price as string (with currency) or add priceCurrency handling.
