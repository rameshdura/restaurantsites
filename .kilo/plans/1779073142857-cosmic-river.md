## Goal

1. Create `apps/web/restaurants/<slug>/images/` folder per restaurant  
2. Copy each restaurant's referenced local image files into that folder  
3. Create a `getImageSrc(slug, url)` helper that resolves image URLs to the correct per-restaurant path  
4. Update code that **constructs/resolves** image URLs from `data.json` to use `getImageSrc()` — data.json itself stays as-is

### Path Resolution Strategy (helper)

```
getImageSrc(slug, rawUrl):
  empty / undefined → ""
  starts with "http"  → rawUrl                         (pass-through external URL)
  starts with "/restaurants/" → rawUrl                (already resolved)
  starts with "/images/" or "/assets/" → 
    /restaurants/<slug>/images/<path-without-leading-slash>
                                                    (serve from restaurant-specific location)

  Example: /images/team/chef.png  → /restaurants/ramen-taro/images/images/team/chef.png
           (slices off the leading /, re-prepends the restaurant URL prefix)
```

This maps to physical files at:
```
public/images/restaurants/ramen-taro/images/team/chef.png
```
which Next.js serves at URL: `/images/restaurants/ramen-taro/images/team/chef.png`

The data.json string `/images/team/chef.png` is passed through `getImageSrc()` at render time, which prepends the restaurant-specific path. Only the **LLM-executed code** (not the data files) uses `getImageSrc` directly — components like `<Hero>`, `<ImageSlider>`, `<TeamSection>` still receive plain URL strings as before.

## Implementation Steps

### Step 1: Add `getImageSrc()` to `apps/web/lib/restaurant.ts`

Add after the existing exports:

```typescript
/**
 * Resolve an image URL stored in data.json to its serving URL.
 *
 * Restaurant-specific local images are mapped to
 *   /images/restaurants/<slug>/images/<path>
 * which is served from   public/images/restaurants/<slug>/images/<path>.
 *
 * External URLs (http/https) are returned as-is.
 * Already-resolved URLs (start with /restaurants/) are returned as-is.
 */
export function getImageSrc(slug: string, url: string | undefined | null): string {
  if (!url) return "";
  if (url.startsWith("/restaurants/")) return url;          // already resolved
  if (/^https?:\/\//i.test(url)) return url;                // external URL
  
  // Local image path: /images/team/chef.png → /images/restaurants/<slug>/images/images/team/chef.png
  // Strip leading / and prepend restaurant prefix
  const trimmed = url.startsWith("/") ? url.slice(1) : url;
  return `/images/restaurants/${slug}/images/${trimmed}`;
}
```

Why the `images/images/` segment? The data.json uses flat paths like `/images/team/chef.png` (relative to `public/images/`). After copying files into `public/images/restaurants/<slug>/images/team/chef.png`, the serving URL must be `/images/restaurants/<slug>/images/team/chef.png`. The `public/`-root segment `images/` is the server root; the `restaurants/<slug>/images/` part preserves the original relative structure within that folder.

### Step 4: Update `normaliseBlockSchema()` in `restaurant.ts` — use `getImageSrc`

The `normaliseBlockSchema` function currently maps `data.team` → `data.about.team` verbatim,
which means `m.image` (e.g. `/images/team/chef.png`) passes through unchanged.

```typescript
// Before
team: data.team?.map((m) => ({
  name: m.name, role: m.role,
  image: m.image,   // raw data.json path — not resolved
  bio: m.bio, social: m.social,
})),

// After
team: data.team?.map((m) => ({
  name: m.name, role: m.role,
  image: getImageSrc(data.uid || "", m.image),  // resolved for correct restaurant
  bio: m.bio, social: m.social,
})),
```

The `team` images flow into `TeamSection` and `about.team` via the `data.about` field set by `normaliseBlockSchema`.

## Why the resolved URL looks like `/images/restaurants/<slug>/images/...`

The double `images/` segments come from two separate layers:
1. `public/` root (served at URL prefix `/`)  
2. `images/restaurants/<slug>/images/` (appended sub-path under `public/images/`)

For example:
```
public/images/restaurants/ramen-taro/images/team/chef.png
                      ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                      — URL prefix /images/ comes from public/images/ root
                        <slug>/images/team/chef.png comes from restaurant folder tree
```

This ensures each restaurant has its own isolated image folder, and the URL for any restaurant
image can be reconstructed deterministically from just `slug` + original `data.json` path.
